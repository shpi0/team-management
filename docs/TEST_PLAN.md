# Team Balancer — план автоматического тестирования (Playwright)

> Детальный план покрытия тест-кейсами для прогона **Playwright-агентом**. Рассчитан на то,
> что агент сам пишет/исполняет шаги; здесь — что проверять, как настроить окружение,
> какие селекторы использовать и где подводные камни. Связано со [SPEC.md](SPEC.md).

---

## 1. Цель и охват

Проверить, что приложение `index.html`:
- корректно грузится и рендерит доску из сохранённого/демо-состояния;
- сохраняет и восстанавливает данные (localStorage + JSON импорт/экспорт);
- даёт корректный CRUD сотрудников, команд, справочников;
- правильно переносит аллокации перетаскиванием и считает метрики/флаги баланса;
- поддерживает дробную аллокацию, снимки/сравнение, undo/redo, фильтры, печать/PNG;
- устойчив к крайним случаям.

Вне охвата: визуальный пиксель-перфект, кросс-браузерный PNG (проверяется лишь факт фолбэка),
производительность на тысячах записей.

## 2. Тестовое окружение и стратегии

### 2.1. Запуск
- Раздать каталог: `python3 -m http.server 8000`, базовый URL `http://localhost:8000/index.html`.
  (HTTP предпочтительнее `file://` — стабильнее download/clipboard/Blob.)
- Браузер: Chromium. Часть кейсов (DnD) проверить также в Firefox по возможности.

### 2.2. Детерминизм состояния — КЛЮЧЕВОЕ
Демо-данные грузятся **только при пустом** `localStorage` (`LS_KEY="team-balancer-v1"`).
Поэтому каждый тест должен стартовать с известного состояния. Два подхода:

**A. Чистый старт на демо-данных:**
```js
await page.goto(URL);
await page.evaluate(() => localStorage.removeItem('team-balancer-v1'));
await page.reload();           // boot → seed()
```

**B. Инъекция фикстуры (рекомендуется для расчётных кейсов):**
```js
await page.goto(URL);
await page.evaluate((fx) => {
  localStorage.setItem('team-balancer-v1', JSON.stringify(fx));
}, FIXTURE);                    // FIXTURE = {state, snapshots:[], baselineId:null, uidCounter:1000}
await page.reload();
```
Фикстуры с заранее известными грейдами/долями позволяют точно проверять метрики и флаги.

### 2.3. Доступ к внутреннему состоянию
Глобальные переменные объявлены через `let` в IIFE-подобном скоупе скрипта и **не**
экспортированы в `window` (кроме `closeModal`). Поэтому проверять состояние следует через:
- DOM (предпочтительно — это и есть наблюдаемое поведение), и/или
- `localStorage` после действия:
  `JSON.parse(await page.evaluate(() => localStorage.getItem('team-balancer-v1')))`.

### 2.4. HTML5 Drag & Drop — особый случай
DnD — нативный (листенеры на `document`, состояние в переменной `dragData`). Стандартный
`locator.dragTo()` Playwright по нативному HTML5 DnD **часто не срабатывает**. Использовать
ручной диспатч событий с общим `DataTransfer` (события всплывают до `document`):

```js
async function html5Drag(page, chipSelector, targetTeamSelector) {
  await page.evaluate(({c, t}) => {
    const src = document.querySelector(c);
    const tgt = document.querySelector(t);      // .team-элемент целевой команды
    const dt = new DataTransfer();
    const fire = (el, type) => el.dispatchEvent(
      new DragEvent(type, { bubbles: true, cancelable: true, dataTransfer: dt }));
    fire(src, 'dragstart');                      // обработчик читает chip.dataset → dragData
    fire(tgt, 'dragover');
    fire(tgt, 'drop');                           // → moveAllocation(...)
    fire(src, 'dragend');
  }, { c: chipSelector, t: targetTeamSelector });
}
```
Селектор источника — конкретная карточка `.chip[data-person-id="…"][data-team-id="…"]`,
цель — `.team[data-team-id="…"]`. Резервный способ проверки логики переноса — если диспатч
не сработал в среде — отдельно зафиксировать как баг (не обходить мышиной эмуляцией молча).

### 2.5. Модалки, тосты, диалоги
- Модалки рендерятся в `#modalRoot` как `.overlay > .modal`. Закрытие: `Esc`, клик по оверлею,
  кнопки «Отмена»/«Готово»/«Закрыть».
- Тосты — в `#toastRoot .toast`, живут ~2.2 c. Проверять `toContainText` сразу после действия.
- `confirm()` (удаление команды с людьми, восстановление снимка) — перехватывать:
  `page.on('dialog', d => d.accept())` (или `dismiss()` для проверки отмены).
- Скачивание файлов (JSON/PNG) — `page.waitForEvent('download')`.

## 5. Готовые хелперы (для агента)

```js
const URL = 'http://localhost:8000/index.html';
const LS = 'team-balancer-v1';

const lsGet   = (page) => page.evaluate(k => JSON.parse(localStorage.getItem(k)||'null'), LS);
const lsSet   = (page, fx) => page.evaluate(([k,v]) => localStorage.setItem(k, JSON.stringify(v)), [LS, fx]);
const lsClear = (page) => page.evaluate(k => localStorage.removeItem(k), LS);

async function bootDemo(page){ await page.goto(URL); await lsClear(page); await page.reload(); }
async function bootFixture(page, fx){ await page.goto(URL); await lsSet(page, fx); await page.reload(); }

// см. §2.4 — нативный HTML5 DnD
async function html5Drag(page, chipSel, teamSel){
  await page.evaluate(({c,t})=>{
    const src=document.querySelector(c), tgt=document.querySelector(t), dt=new DataTransfer();
    const f=(el,ty)=>el.dispatchEvent(new DragEvent(ty,{bubbles:true,cancelable:true,dataTransfer:dt}));
    f(src,'dragstart'); f(tgt,'dragover'); f(tgt,'drop'); f(src,'dragend');
  },{c:chipSel,t:teamSel});
}

// перехват confirm
page.on('dialog', d => d.accept());        // или d.dismiss() в кейсах отмены

// проверка отсутствия JS-ошибок
const errors=[]; page.on('pageerror', e=>errors.push(e)); // assert errors.length===0
```

## 6. Критерии приёмки прогона

- Все P0 — зелёные (блокеры релиза).
- P1 — зелёные или с заведёнными багами с воспроизведением.
- Ни в одном основном сценарии нет `pageerror`.
- Кейсы DnD (TS-5, TS-6.1) подтверждены реальным переносом аллокаций в LS, а не только DOM.
- Расчётные кейсы (TS-7) проверены на фикстурах с заранее посчитанными ожиданиями.

### 6.1. Representative-покрытие (не отдельные тесты в раннере)

Следующие кейсы плана **намеренно** покрыты представителями своей группы, а не отдельными
тестами в `docs/run-tests.js`. Прохождение раннера (его счётчик) **не** означает дословное
покрытие каждого ID плана — это осознанный объём, а не пробел:

| ID | Покрыто представителем |
|----|------------------------|
| TC-7.4 (флаг по доле подрядчиков) | TC-7.3 (механика z-флага идентична, другая метрика) |
| TC-7.6 (команд < 2 → std=0) | TC-7.5 (та же ветка `std=0`) |
| TC-9.5 (серия undo) | TC-9.1/9.2 (одиночные undo/redo) |
| TC-10.5 (комбинация фильтров) | TC-10.1–10.4 (отдельные фильтры) |
| TC-11.5 (rename/delete локаций) | TC-11.1–11.4 (те же операции над ролями) |
| TC-14.4 (длинные имена, 12+ команд) | ручная/визуальная проверка (лейаут, не логика) |

При необходимости перевести любой из них в обязательный automated — добавить отдельный `T(...)`
в раннер и убрать строку отсюда.
