## Findings v2

### V2-F1 (P1) — `normalizeDoc` не устраняет дубли безопасных id

Файлы:

- `index.html:311-317` — нормализация команд.
- `index.html:322-333` — нормализация людей.
- `index.html:341-348` — нормализация снимков.

Проблема:

`normalizeDoc` регенерирует id только если он не проходит `SAFE_ID`, но не проверяет уникальность безопасных id.

Примеры:

```js
let id = asStr(t.id);
if(!SAFE_ID.test(id)){ ... }
return { id, ... };
```

```js
let id = asStr(p.id);
if(!SAFE_ID.test(id)) id = uid("p");
```

Если импорт содержит две команды с `id: "tA"`, обе останутся с одинаковым id. Это ломает модель данных, потому что дальше код предполагает уникальные id.

Последствия:

- две команды получают одинаковый `data-team-id`;
- `teamMetrics(team.id)` посчитает одни и те же аллокации для обеих команд;
- удаление команды по id удалит обе команды;
- rename/color/edit через `find` попадут в первую найденную запись;
- DnD и compare по людям с одинаковым `person.id` будут неоднозначны.

Рекомендация:

- В `normalizeDoc` добавить `seenTeamIds` и `seenPersonIds`.
- Если id безопасный, но уже занят, регенерировать новый id.
- Для команд поддерживать `idMap`, чтобы аллокации старого duplicate id были перепривязаны по выбранной политике.
- Для людей с duplicate id также регенерировать id.
- В `normalizeSnapshots` добавить `seenSnapshotIds`.
- Добавить тест `TC-2.9`: импорт с duplicate team/person/snapshot ids, проверка уникальности после нормализации и отсутствия `pageerror`.

### V2-F2 (P1/P2) — ремап unsafe id между `state` и `snapshots` несогласован

Файлы:

- `index.html:304-339` — `normalizeDoc`.
- `index.html:341-348` — `normalizeSnapshots`.
- `index.html:464-470` — compare baseline.
- `index.html:604-608` — `.moved-in` по `person.id`.

Проблема:

При полном импорте `state` и каждый snapshot нормализуются независимо:

```js
state = normalizeDoc(doc);
snapshots = normalizeSnapshots(snaps);
```

`normalizeSnapshots()` внутри вызывает `normalizeDoc(s.doc)`. Если в полном импортируемом объекте одни и те же небезопасные team/person ids присутствуют и в `state`, и в `snapshot.doc`, они могут получить разные новые id. После этого compare-режим будет видеть текущий документ и baseline как разные сущности.

Последствия:

- все карточки могут подсвечиваться как `moved-in`, потому что `baseMap.get(p.id)` не найдёт старый person id;
- team diff может не строиться для переименованных/перегенерированных team ids, потому что `baseStatsByTeam[team.id]` не совпадёт;
- восстановление снимка может заменить state на документ с другим набором новых id, что меняет идентичность сущностей без пользовательского действия.

Рекомендация:

- Для полного импорта нормализовать `state` и `snapshots` согласованно.
- Возможный подход: построить общий `idMap` для unsafe/duplicate ids на уровне импортного пакета и применять его к `state` и всем `snapshot.doc`.
- Минимально: если согласованный ремап слишком дорогой, явно документировать, что импорт старых/грязных snapshot ids может потерять точность compare, и добавить тест на текущую политику.
- Лучше добавить тест `TC-12.8`: импорт полного объекта со snapshot/base, где unsafe ids одинаковы в state и snapshot, включить compare и проверить, что неизменённые люди не подсвечиваются как moved-in.
