# Team Balancer — спецификация проекта

> Полная локальная спецификация. Источник истины для продолжения работы в любой сессии.
> Дата создания: 2026-06-30. Версия документа: 1.0. Версия приложения: 1.

---

## 1. Назначение

Инструмент ИТ-лидера кластера для **визуализации и балансировки команд** продуктовой
разработки. Позволяет видеть состав команд, перетаскивать участников между ними и сразу
наблюдать, где баланс по сеньорности и доле подрядчиков нарушен, а где соблюдён.

Основной пользователь: руководитель кластера из ~10 команд. Сценарии: ревизия состава,
планирование перестановок («что будет, если перенести X в команду Y»), подготовка
отчётов/презентаций по структуре кластера.

---

## 2. Жёсткие ограничения (НЕ нарушать при доработках)

1. **Закрытый корпоративный финтех-контур.** Приложение должно запускаться без интернета,
   установки и админ-прав: один HTML-файл, открываемый в браузере. **Запрещены** CDN,
   npm-зависимости, шаги сборки, внешние сетевые запросы, веб-шрифты с CDN, аналитика.
   Любая новая функциональность — только vanilla JS/CSS внутри одного файла.
2. **Приватность данных.** В данных — реальные ФИО сотрудников. Ничего не должно покидать
   машину пользователя. Хранение только локальное (`localStorage`) + ручной экспорт/импорт
   JSON-файла. Никаких автозагрузок на сервер.
3. **Без бэкенда.** Чистое клиентское приложение.

Эти ограничения — продуктовые, не технические капризы. Нарушение делает инструмент
непригодным для целевого контура.

## 3. Технологическая архитектура

- **Единственный артефакт:** `/workspace/index.html` (~900 строк). Внутри три секции:
  `<style>` (тема, лейаут, печать), разметка шапки/доски, `<script>` (вся логика).
- **Без фреймворков.** Состояние в модульных переменных, ручной рендер через `innerHTML` +
  делегирование событий. Это сознательный выбор ради нулевых зависимостей.
- **Запуск:** двойной клик по файлу (`file://`) либо раздача любым статик-сервером.
  Сборки/транспиляции нет.
- **Поддержка браузеров:** современные Chromium/Firefox (корпоративные Edge/Chrome). ES2020+
  (опциональные цепочки, `??`), `URL.createObjectURL`, `FileReader`, нативный HTML5 DnD.

### 3.1. Организация кода (внутри `<script>`)

| Блок | Назначение |
|------|-----------|
| Константы | `LS_KEY`, `GRADE_MIN=7`, `GRADE_MAX=12`, `Z_WARN=0.85`, `Z_BAD=1.6` |
| `state / snapshots / baselineId / ui / history` | Глобальное состояние приложения |
| `seed()` | Демо-данные при первом запуске (4 команды, 15 человек) |
| `persist() / load()` | localStorage |
| `pushHistory / commit / undo / redo` | История изменений документа |
| `teamMetrics / clusterStats / zFlag` | Доменная логика баланса |
| `render / renderAnalytics / renderTeam / renderChip` | Рендер |
| DnD-листенеры + `moveAllocation` | Перетаскивание |
| `personModal / settingsModal / snapshotModal` | Модальные окна |
| `downloadJSON / exportAll / importFile / exportPNG` | Импорт-экспорт |
| Wire-up в конце | Привязка кнопок, делегирование по доске, горячие клавиши, `boot` |

**Конвенция `commit()`:** любая мутация документа `state` должна заканчиваться вызовом
`commit()` — он делает `pushHistory()` (для undo), `persist()` (в localStorage) и `render()`.
Изменения только UI/фильтров — вызывают `render()` напрямую без `commit()`.

## 4. Модель данных

```js
state = {
  meta: { name: String, version: 1 },
  roles:     [String],          // упорядоченный справочник ролей, редактируемый
  roleColors: { [role]: "#rrggbb" }, // цвет квадрата грейда по роли (необязат.; иначе палитра по индексу)
  locations: [String],          // справочник локаций, редактируемый
  projects:  [String],          // справочник проектов (для типа ставки), редактируемый
  teams: [
    { id: "t…", name: String, color: "#rrggbb" }   // порядок в массиве = порядок на доске (drag-reorder)
  ],
  people: [                     // «люди» включают и ставки (вакансии) — isVacancy
    {
      id: "p…",
      name: String,             // у ставки (isVacancy) — "" (имя не указывается)
      role: String,             // одно из roles (или произвольное)
      grade: Number | null,     // 7..12; null у подрядчиков; у ставки опционален (может быть null)
      isContractor: Boolean,    // true => grade=null; у ставки всегда false
      isVacancy: Boolean,       // ставка — открытая штатная позиция без сотрудника
      location: String,         // одно из locations; у ставки — ""
      project: String,          // только у ставки: проект из projects ("" = обычная ставка)
      expiry:  String,          // только у ставки: "YYYY-MM-DD" срок действия ("" = бессрочная)
      tags: [String],           // уникальные навыки; только отображение, в аналитике (теги) учитываются
      comment: String,          // текстовая заметка; видна ТОЛЬКО в модалке, не на доске
      allocations: [            // ДРОБНАЯ аллокация: сотрудник может быть в нескольких командах
        { teamId: "t…", fte: Number }   // fte > 0, обычно 0..1; сумма по человеку не ограничена жёстко
      ]
    }
  ]
}

snapshots = [
  { id: "s…", name: String, createdAt: "YYYY-MM-DD HH:MM", doc: <глубокая копия state> }
]
baselineId = "s…" | null        // снимок, выбранный как база для сравнения as-is/to-be

// UI-состояние (частично персистентно): ui.hideGrades — глобальное скрытие грейдов («глаз»),
// сохраняется в localStorage отдельно от документа.
```

### 4.1. Ключевые инварианты

- `id` генерируются `uid(prefix)` = `prefix + base36(timestamp) + base36(counter)`; уникальны.
- Сотрудник присутствует в команде столько раз, сколько у него аллокаций на эту команду
  (нормально — одна). Каждая аллокация рендерится **отдельной карточкой**.
- Грейд жёстко клампится в диапазон `7..12` при сохранении (`clampGrade`).
- При `isContractor === true` поле `grade === null`, и грейд исключается из расчётов.
- Аллокации с `fte <= 0` отбрасываются при сохранении; у человека всегда ≥1 аллокация
  (иначе он удаляется, например при удалении его единственной команды).

## 5. Хранение и форматы обмена

- **localStorage**, ключ `team-balancer-v1`. Значение:
  `{ state, snapshots, baselineId, ui:{}, uidCounter }`. Пишется на каждом `persist()`
  (debounce не нужен — операции редкие, объём мал). При ошибке (квота/приватный режим)
  тихо игнорируется — экспорт в файл остаётся доступным.
- **Экспорт «JSON» (кнопка ⬇):** скачивает `{ state, snapshots, baselineId, version:1 }`
  именем `<имя кластера>.json`.
- **Импорт (кнопка ⬆ → file input):** принимает либо полный объект (`{state,…}`), либо
  «голый» документ (`{teams, people, …}`). После импорта история сбрасывается, в тосте —
  сводка (сколько команд/людей загружено, сколько записей отброшено).
- **Экспорт снимка** (в окне «Снимки», ⬇): скачивает только `doc` выбранного снимка.

### 5.1. Нормализация и валидация (импорт и localStorage — недоверенный вход)

И `load()`, и `importFile()` пропускают документ через `normalizeDoc()` (а снимки — через
`normalizeSnapshots()`) перед использованием. Политика:

- **Обязательные поля гарантируются:** `meta`, `roles`, `locations`, `teams`, `people` всегда
  существуют (отсутствующие → пустые/дефолтные). Не-массивы заменяются на массивы.
- **Справочники `roles`/`locations` дедуплицируются** с предварительным `trim` (первое
  вхождение; пустые/whitespace-only/не-строки отброшены) — как UI add/rename. Поля
  `people[].role` и `people[].location` тоже `trim`-ятся, чтобы значения у сотрудников
  совпадали со справочником (без `" Backend "` vs `"Backend"`). Импорт/localStorage больше
  не оставляют обходной путь к «почти-дублям».
- **Команды:** `id` обязан совпадать с `^[A-Za-z0-9_-]+$` **и быть уникальным в документе**,
  иначе перегенерируется (`uid`); ссылки в аллокациях на небезопасный id ремапятся. Для
  дубликата безопасного id первая команда сохраняет id (аллокации на него уходят к ней),
  последующие получают новый id. `color` обязан быть `^#[0-9a-fA-F]{6}$`, иначе `#4f8cff`.
- **Справочник `projects`** дедуплицируется с `trim` (как roles/locations).
- **Люди:** `id` чинибельный/перегенерируемый **и уникальный** (дубль → новый id);
  `grade` парсится и клампится в `7..12`; у подрядчиков `grade=null`;
  `name`/`role`/`location` приводятся к строкам (role/location — c `trim`).
- **Ставки (`isVacancy`):** `isContractor` форсится в `false`, `name`/`location` → `""`,
  `grade` опционален (пусто → `null`); `project` — **только из справочника `projects`** (trim;
  неизвестный → `""`, чтобы доска и модалка не расходились); `expiry` — **календарно валидная**
  дата `YYYY-MM-DD` (`validDateISO`; `2026-99-99` → `""`), иначе `""`. У не-ставок `project`/`expiry` → `""`.
- **`comment`** приводится к строке.
- **Теги людей:** `tags` приводится к массиву строк (trim, отбрасывание пустых, дедупликация,
  лимиты ≤24 симв./≤12 шт.); не-массив → `[]`.
- **`roleColors`:** оставляются только записи с валидным hex6; прочие отбрасываются.
- **Снимки:** `id` санируется и **дедуплицируется** в пределах списка.
- **Аллокации:** оставляются только с `fte>0` (парсится из строки/числа) **и** существующим
  `teamId`; `fte`-строки/`NaN`/отрицательные — отбрасываются.
- **Люди без валидных аллокаций отбрасываются** (политика «не показывать осиротевших»).
- **`baselineId`** оставляется, только если указывает на существующий снимок, иначе `null`;
  `ui.compare` сбрасывается.

Это закрывает и инъекцию через импортированные `id`/`color` в HTML-атрибуты/`style`
(значения санируются у источника, до рендера).

**Граница (известное ограничение).** При **полном** импорте `state` и каждый `snapshot.doc`
нормализуются независимо. Данные, созданные приложением, всегда имеют безопасные уникальные id
(`normalizeDoc` для них идемпотентен), поэтому сравнение as-is/to-be после round-trip
экспорт/импорт работает корректно (тест TC-12.8). Однако **рукотворный** импорт, где одни и те
же *небезопасные/дублирующиеся* id присутствуют и в `state`, и в `snapshot.doc`, может получить
рассогласованные новые id — тогда compare с таким снимком потеряет точность (всё подсветится
как `moved-in`). Общий межрдокументный ремап намеренно не делается: реальные данные его не
требуют, а он усложнил бы нормальный путь compare. Чинибельно при реальной потребности (бэклог §10).

## 6. Доменная логика баланса

Режим: **сравнение команд между собой** (не целевые эталоны). Подсвечиваются команды-выбросы.

### 6.1. Метрики команды — `teamMetrics(teamId)`

Идёт по всем аллокациям всех людей на эту команду:

| Поле | Формула |
|------|---------|
| `fte` | Σ `fte` всех аллокаций на команду (без ставок) |
| `head` | число аллокаций укомплектованных (≈ число человек, без ставок) |
| `avgGrade` | Σ(`grade·fte`) / Σ`fte` по **штатным с грейдом**; `null`, если таких нет |
| `contrShare` | (Σ`fte` подрядчиков) / `fte`; 0 при пустой команде |
| `locs` | распределение `fte` по локациям |
| `vacancies` / `vacancyFte` | число ставок и их суммарный FTE — отдельно |

> Грейд взвешивается по FTE: человек на 0.5 FTE даёт вес 0.5. Подрядчики в средний грейд
> не входят (у них нет грейда), но входят в `contrShare` и `fte`. **Ставки (`isVacancy`)
> полностью исключены** из `fte/head/avgGrade/contrShare/locs` и считаются отдельно.

### 6.2. Статистика кластера — `clusterStats()`

Считает по командам среднее и **популяционное** стандартное отклонение (делёж на N):
- `grade.{mean,std}` — по командам, где `avgGrade != null`;
- `share.{mean,std}` — по всем командам;
- `std` возвращает `0`, если команд < 2.

### 6.3. Флаг выброса — `zFlag(val, ref)`

```
если val == null или ref.std == 0  → { z:0, level:"good" }
иначе z = |(val - ref.mean) / ref.std|
level = z > 1.6 (Z_BAD)  → "bad"  (красный)
        z > 0.85 (Z_WARN)→ "warn" (жёлтый)
        иначе            → "good" (зелёный)
```

Применяется отдельно к `avgGrade` (vs `grade`) и к `contrShare` (vs `share`) каждой команды.
Цвет точки/полоски: `bad=#e5484d`, `warn=#e0a516`, `good=#2fbf71`.

### 6.4. Сводная аналитика — `renderAnalytics()`

Карточки: средний грейд по кластеру (взвешен по **штатному grade-FTE**, т.е. сумме FTE только
тех аллокаций, у кого есть грейд — подрядный FTE в вес не входит); доля подрядчиков (%);
суммарный FTE и средний на команду; **«Несбалансированные команды»** (число команд с сильным
отклонением среднего грейда или доли подрядчиков от среднего по кластеру — переименовано из
неясного «Команды-выбросы»); **«Открытые ставки»** (всего + обычных/проектных/временных);
команд/людей; распределение по локациям (бар-чарт); **«Топ тегов»** (бар-чарт по навыкам).

## 7. Функциональные требования (реализовано в v1)

### 7.1. Доска кластера
- Адаптивная сетка карточек-команд (`grid auto-fill, minmax(280px,1fr)`).
- Шапка команды: цвет, редактируемое имя, кнопка удаления; три метрики (средний грейд +
  флаг, % подрядчиков + флаг, FTE/чел.). Под метриками — diff-строка в режиме сравнения.
- Участники **сгруппированы по ролям** (порядок ролей — как в справочнике, неизвестные в
  конец). Заголовок группы: роль, число людей и суммарный FTE, кнопка «＋» (добавить в роль).

### 7.2. Карточка участника (chip)
- Бейдж грейда (число) или «ПД» для подрядчика; имя + бейдж «подряд»; строка «роль · локация»;
  бейдж доли FTE, если `fte < 1`; кнопка ✎ (редактирование, видна по ховеру).
- `draggable`. Атрибуты `data-person-id`, `data-team-id`.

### 7.3. Drag & Drop (перенос аллокации целиком)
- Перетаскивание карточки из команды A в команду B переносит **эту аллокацию целиком**
  (`moveAllocation`). Если у человека уже есть аллокация в B — доли складываются (округление
  до 2 знаков), исходная удаляется. Перенос в ту же команду игнорируется.
- Реализация: нативный HTML5 DnD, листенеры на `document`, состояние перетаскивания в
  модульной переменной `dragData = {personId, fromTeamId}`. Drop-цель — `.team`.
- **Дробление** доли делается не перетаскиванием, а в диалоге редактирования (раздел 7.4).

### 7.4. CRUD сотрудника (`personModal`)
- Поля: имя; статус Штат/Подрядчик (сегмент-переключатель; при «Подрядчик» поле грейда
  блокируется и обнуляется); грейд (number 7..12); роль (select из справочника); локация
  (select); редактор аллокаций — список «команда + доля FTE» с добавлением/удалением строк
  и живым подсчётом суммарного FTE. При суммарном FTE > 1.0 показывается предупреждение
  (⚠, без запрета — перегрузка допускается намеренно, см. §9).
- Валидация: имя по умолчанию «Без имени»; грейд клампится; аллокации с `fte>0`; требуется ≥1.
- Кнопка «Удалить» (только при редактировании). Добавление: «＋ Сотрудник» (шапка),
  «Добавить в команду» (футер команды), «＋» у роли (`quickAdd` → `personModal(null, teamId, role)`,
  режим **создания** с предзаполненными командой и ролью).

### 7.5. CRUD команды
- «＋ Команда» в шапке и «＋ команда» в справочниках — добавляет команду с цветом из палитры.
- Переименование: инлайн-поле в шапке (`data-rename`) и в справочниках (`data-tname`).
- Цвет: `input[type=color]` в справочниках (`data-tcolor`).
- Удаление: ✕ в шапке (`data-delteam`) или в справочниках (`data-trm`) — общий хелпер
  `deleteTeamById`. **Подтверждение, если есть участники ИЛИ открытые ставки** (текст упоминает
  оба); снимает аллокации на эту команду; осиротевшие записи (включая ставки) удаляются.

### 7.6. Справочники (`settingsModal`)
- Роли и локации: добавление, инлайн-переименование, перемещение ↑/↓, удаление.
- **Переименование = настоящее переименование (продуктовое решение):** мигрирует значение у всех
  сотрудников (`people[].role` / `people[].location`), а для роли — переносит и явный цвет
  (`roleColors`). Так справочник и сотрудники остаются согласованы, без «осиротевших» значений.
- **Дубли при переименовании запрещены:** rename в уже существующее имя отклоняется (поле
  возвращает прежнее значение + toast). Добавление дубля также игнорируется.
- **Удаление** из справочника **не** меняет значение у существующих людей (они сохранят прежнюю
  строку, которая станет «несправочной»); для роли удаляется её запись в `roleColors`.
- Команды: список с именем, цветом, удалением; добавление.

### 7.7. Снимки и сравнение as-is / to-be (`snapshotModal`)
- «Сохранить текущее» → снимок с именем и временной меткой (`doc` = копия `state`).
- По снимку: «сделать базой»/«снять базу», «восстановить» (заменяет текущее, с подтверждением,
  обратимо через Undo), экспорт ⬇, удаление ✕.
- Когда база выбрана — в верхней панели появляется галочка «сравнить с базой». При включении:
  переехавшие участники в новых для них командах подсвечиваются зелёной рамкой (`moved-in`),
  по командам в шапке выводятся дельты среднего грейда и FTE.

### 7.8. Undo / Redo
- `commit()` кладёт глубокую копию документа в `history` (до 60 шагов). Undo/Redo двигают
  указатель `hpos`. Горячие клавиши: `Ctrl/Cmd+Z`, `Ctrl/Cmd+Y` или `Ctrl/Cmd+Shift+Z`.
  Не срабатывают, когда фокус в input/select/textarea.
- **Область охвата (продуктовое решение):** история хранит только документ `state` (состав
  команд, сотрудники, справочники, имя кластера). Операции со **снимками** (создание/удаление),
  выбор **baseline** и флаг `ui.compare` под undo/redo **не** попадают — это операции над
  сценариями, а не над текущим составом. При необходимости расширить — см. бэклог §10.

### 7.9. Поиск и фильтры
- **Поиск** (`matchFilter`) по имени, роли, проекту, тегам и слову «вакансия» (для ставок);
  комментарий намеренно не ищется (он скрыт с доски).
- **Фильтры** по роли, локации, статусу. Статус: «Все», «Только штат» (реальные штатные — **без
  ставок и подрядчиков**), «Только подрядчики», «Только ставки» (вакансии).
- По умолчанию несовпавшие приглушаются (`dim`), совпавшие подсвечиваются (`hit`); чекбокс
  «скрывать несовпадающих» прячет их.

### 7.10. Экспорт изображения / печать
- **Печать (🖨)** → `window.print()` со специальным print-CSS (3 колонки, светлая тема).
  Надёжный путь для PDF/отчётов.
- **PNG (🖼)** → best-effort через SVG `foreignObject` → canvas → PNG (масштаб ×2). При сбое
  (ограничения браузера на foreignObject/taint) автоматически открывается печать.

### 7.11. Скрытие грейдов («глаз», `#gradeEye`)
- Кнопка-переключатель в шапке. `ui.hideGrades` глобально маскирует **индивидуальные** грейды
  на карточках (квадрат показывает `•••`, цвет роли сохраняется). Подрядчики (`ПД`) не затрагиваются.
- Агрегаты (средний грейд команды/кластера) **не** маскируются — это аналитическое ядро и
  значительно менее чувствительны.
- Состояние персистентно (localStorage, отдельно от документа), переживает reload.
- **Это визуальная маскировка доски (для демонстрации/скриншота/шеринга экрана), а НЕ
  privacy/security-граница.** Значения грейдов остаются доступны: в `localStorage`, в JSON-экспорте,
  в модалке редактирования сотрудника и косвенно через агрегаты. Полноценный privacy-режим
  (маскирование экспорта/модалки/агрегатов) — отдельная задача, см. бэклог §10.

### 7.12. Теги сотрудников
- Поле в модалке (`#m_tags`, через запятую). Хранятся как `person.tags: [String]`.
- Нормализация (`normTags`): trim, отбрасывание пустых, дедуп, **лимиты** — длина тега
  ≤ `MAX_TAG_LEN` (24), количество ≤ `MAX_TAGS` (12). CSS-клэмп пилюль (`max-width`, ellipsis)
  защищает плотность доски от очень длинных токенов.
- Отображаются на карточке пилюлями (`.tag-pill`) под строкой «роль · локация».
- **В балансе не участвуют** (не влияют на грейд/подрядчиков/выбросы) — только навыки. В сводной
  аналитике есть отдельный блок «Топ тегов» (распределение по навыкам, см. §7.18).

### 7.13. Перестановка команд по доске (drag-reorder)
- Грип `⠿` в шапке команды (`.team-grip[data-teamdrag]`, `draggable`). Перетаскивание команды на
  другую вставляет её на позицию целевой (`reorderTeam`), порядок сохраняется в `state.teams`.
- Отделено от DnD участников: грип-перенос → `teamDrag`, перенос карточки → `dragData`.
  Обработчики `dragstart/over/drop` различают их; перенос команды undo-обратим.

### 7.14. Цвет квадрата грейда по роли
- Квадрат грейда штатного сотрудника красится цветом его роли (`roleColor(role)`): явный цвет из
  справочника (`state.roleColors[role]`) либо детерминированный из палитры по индексу роли.
- Настраивается в «Справочниках» рядом с ролью (`input[type=color]`, `[data-rolecolor]`).
  Цвет применяется **только на `change`** (единая undo-точка + persist); промежуточные `input`
  не мутируют `state` (избегаем грязного состояния без истории/persist).
- Переименование роли переносит цвет **и** мигрирует сотрудников (см. §7.6), так что существующие
  сотрудники сохраняют свой цвет; удаление роли убирает её запись в `roleColors`.
- Подрядчики сохраняют отдельный цвет (`--contractor`) — чтобы статус штат/подряд оставался
  визуально различим.

### 7.15. Ставки (вакансии, `isVacancy`)
- Галка «Это ставка» в модалке (`#m_vacancy`) превращает запись в **открытую штатную позицию без
  сотрудника**: имя/статус/локация скрываются, `isContractor=false`, грейд опционален; показываются
  поля проекта и срока (см. §7.17).
- На карточке: пунктирная рамка (`.chip.vacancy`), подпись «Вакансия», квадрат грейда цвета роли
  (грейд или «—»). В шапке команды — строка «Открытые ставки: N · X FTE» (если есть).
- **Исключены из метрик укомплектованности:** не входят в средний грейд, долю подрядчиков, FTE/чел.,
  `totalHead`, локации. Считаются отдельно (`teamMetrics.vacancies/vacancyFte`, `clusterStats.vacancies`).

### 7.16. Комментарий к сотруднику (`comment`)
- Текстовое поле (`#m_comment`, textarea) в модалке. **Не отображается на доске** — виден только
  при открытии карточки на редактирование. Применимо и к ставкам.

### 7.17. Тип ставки: проект и срок действия
- Справочник **проектов** (`state.projects`) редактируется в «Справочниках»: добавление,
  переименование с миграцией `people[].project`, дубли запрещены. **Удаление проекта снимает его
  со ставок** (`people[].project → ""`) — в отличие от ролей/локаций, чтобы держать строгий
  инвариант `project === "" || projects.includes(project)` (см. V6-F2).
- У ставки: `project` (**только из справочника** — неизвестный сбрасывается при нормализации,
  см. §5.1) и `expiry` (календарно валидная дата или пусто). Обычная ставка = без проекта и без
  срока. На доске чипсы: `▣ проект` (`.vac-chip.proj`) и `⏳ до <дата>` (`.vac-chip.temp`) —
  только если заданы; иначе ничего дополнительно.

### 7.18. Аналитика по ставкам и тегам
- Карточка «Открытые ставки»: всего + разбивка «обычных / проектных / временных» (категории
  «с проектом» и «со сроком» могут пересекаться).
- Карточка «Топ тегов»: распределение по навыкам (бар-чарт, топ-8), учитывает и сотрудников, и ставки.

## 9. Известные ограничения и подводные камни

1. **PNG-экспорт нестабилен между браузерами** (foreignObject может «затейнить» canvas в
   части движков). Поэтому есть автоматический фолбэк на печать. Для гарантированного
   результата — печать в PDF.
2. **Нативный HTML5 DnD** трудно драйвить автоматизацией (см. TEST_PLAN §2.4) — нужен ручной
   диспатч `DragEvent` или вызов `moveAllocation` напрямую.
3. **Демо-данные** грузятся только при пустом `localStorage`. Для детерминизма тестов LS надо
   чистить/инициализировать явно.
4. **Сумма FTE по человеку не ограничена** сверху (можно сделать >1) — намеренно, для гибкости
   сценариев; запрета нет, но при >1.0 в модалке показывается предупреждение (⚠).
5. **Роли/локации в справочнике и у людей не синхронизируются** при удалении — намеренно.
6. **Округление** долей при слиянии аллокаций — до 2 знаков.

## 10. Возможные расширения (бэклог, не реализовано)

- Целевые эталоны по командам (нужные роли/размер/бюджет грейдов) как второй режим баланса.
- Цветовая раскраска ролей; настраиваемые пороги выброса в UI.
- Подсветка дыр в покрытии ролей (как метрика баланса).
- Диалог дробления прямо при перетаскивании (Shift-drag).
- Множественный выбор/массовые перестановки.
- Тёмная/светлая тема, локали.
- Расширение undo/redo на снимки/baseline (`{state, snapshots, baselineId}`) — см. §7.8.
- Согласованный межрдокументный ремап id при полном импорте рукотворных пакетов с грязными
  id (общий `idMap` для `state` и всех `snapshot.doc`) — см. ограничение в §5.1.
- Полноценный privacy-режим для грейдов (маскирование JSON-экспорта, модалки и агрегатов), а не
  только визуальная маскировка доски — см. §7.11.
- Merge ролей (объединение двух ролей с миграцией сотрудников) как явная операция справочника.

## 11. Как продолжать работу

1. Открыть `/workspace/index.html` в браузере — это и приложение, и весь исходник.
2. Любая правка логики → искать соответствующий блок по карте из §3.1.
3. После правок мутаций убедиться, что путь заканчивается `commit()`.
4. Проверить синтаксис: извлечь `<script>` и `node --check` (см. CLAUDE.md).
5. Прогнать тест-план `docs/TEST_PLAN.md` (Playwright-агент).
6. Связанные заметки памяти: `team_balancer.md`, профиль пользователя `user_profile.md`.
