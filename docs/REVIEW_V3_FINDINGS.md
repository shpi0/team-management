# Review v3 findings

Дата: 2026-06-30.

Контекст: после добавления новых фич были повторно проверены спецификация, реализация и автотесты. Полный прогон `node docs\run-tests.js` проходит: `88/88 passed`. Дополнительно вручную проверены новые сценарии через Playwright.

## Summary

Новые фичи в базовом happy path работают: скрытие грейдов, теги сотрудников, drag-reorder команд, цвета грейда по роли, сохранение в localStorage. Перестановка команд фактически undo/redo-обратима и переживает reload.

Основные риски сейчас сосредоточены вокруг справочника ролей: `roleColors` привязан к имени роли, но переименование роли не мигрирует сотрудников и допускает дубли. Это создает неоднозначное и местами неожиданное поведение, которое текущие тесты не покрывают.

## Findings

### F1 (P1): переименование роли переносит цвет, но не переносит сотрудников

**Где:**

- `index.html:268` - `roleColor(role)` ищет явный цвет по `state.roleColors[role]`, иначе берет палитру по индексу роли.
- `index.html:907` - обработчик `[data-roleedit]` при переименовании переносит `roleColors[old]` в `roleColors[nv]` и удаляет старую запись.
- `index.html:887` - подсказка в справочниках говорит, что переименование роли не трогает сотрудников.
- `docs/SPEC.md:306` - цвет грейда штатного сотрудника определяется его ролью.
- `docs/SPEC.md:309` - переименование роли переносит цвет.

**Что происходит:**

Если задать явный цвет роли `Backend`, а затем переименовать роль в `Backend2`, сотрудники с ролью `Backend` остаются с прежним текстовым значением роли. При этом цвет из `roleColors.Backend` удаляется и переносится в `roleColors.Backend2`. В результате существующие сотрудники `Backend` теряют явно заданный цвет и получают fallback-цвет.

**Почему это риск:**

Сейчас одновременно верны два правила, которые конфликтуют:

- переименование роли не меняет сотрудников;
- цвет роли переносится на новое имя.

Для пользователя это выглядит как неожиданная потеря цвета у существующих сотрудников. Для модели это особенно заметно, потому что `roleColors` хранится по имени роли, а не по стабильному id.

**Рекомендация:**

Нужно явно выбрать продуктовую семантику:

1. Если rename роли означает изменение справочника без миграции сотрудников, то старый цвет нужно оставлять или копировать для старого значения, чтобы существующие сотрудники не меняли внешний вид.
2. Если rename роли означает настоящее переименование роли, то нужно мигрировать `people[].role` со старого значения на новое.
3. Если старое значение после rename считается "несправочной ролью", это нужно явно описать в SPEC и тестах, включая ожидаемую смену цвета.

**Тесты:**

Добавить сценарий:

- задать `roleColors.Backend = "#ff0000"`;
- переименовать `Backend` в `Backend2`;
- проверить выбранную продуктовую семантику для сотрудников со старой ролью и для `state.roleColors`.

### F2 (P1): переименование роли допускает дубли

**Где:**

- `index.html:907` - `[data-roleedit]` принимает любое непустое значение и не проверяет уникальность.
- `index.html:892` - добавление через `[data-add="roles"]` проверяет `includes`, то есть дубли при добавлении запрещены.

**Что происходит:**

Роль можно переименовать в уже существующую. Например, `Backend` можно переименовать в `Frontend`; после этого `state.roles` содержит два `Frontend`.

**Почему это риск:**

Дубли ломают однозначность справочника:

- `roleColors` хранится по имени роли, поэтому две одинаковые роли не могут иметь разные явные цвета;
- fallback-палитра зависит от `state.roles.indexOf(role)`, поэтому используется первый индекс, а второй дубль практически неразличим;
- UI редактирования сотрудника получает две одинаковые опции;
- дальнейшее удаление/перемещение одной из ролей становится неоднозначным для пользователя.

**Рекомендация:**

Запретить rename в уже существующее имя, если это не та же самая строка в текущем индексе. При конфликте лучше возвращать старое значение и показывать toast. Альтернативный вариант - явно реализовать merge ролей с миграцией сотрудников, но это более крупная продуктовая операция.

**Тесты:**

Добавить сценарий:

- открыть справочники;
- попытаться переименовать первую роль во вторую;
- проверить, что `state.roles` остался уникальным и UI не создал дубль.

### F3 (P2): TC-15.3 заявляет undo/reload, но тест это не доказывает

**Где:**

- `docs/TEST_PLAN.md:299` - `TC-15.3` требует persistence и undo-обратимость перестановки команд.
- `docs/run-tests.js:1058` - тест проверяет порядок в DOM и `state.teams` в localStorage, но не нажимает undo/redo и не делает reload.

**Что проверено вручную:**

Реализация работает: после drag-reorder порядок меняется, `#undo` возвращает старый порядок, `#redo` снова применяет новый порядок, reload сохраняет порядок.

**Почему это риск:**

Сейчас тестовый отчет может создавать ложное ощущение, что undo/reload покрыты автотестом. Будущая регрессия в `commit()`/history для reorder может пройти незамеченной.

**Рекомендация:**

Усилить `TC-15.3`:

- после reorder нажать `#undo` и проверить исходный порядок;
- нажать `#redo` и проверить новый порядок;
- сделать reload и проверить сохраненный порядок.

### F4 (P2): недостаточно покрыта жизненная логика `roleColors`

**Где:**

- `docs/run-tests.js:1073` - `TC-15.4` проверяет разные fallback-цвета для разных ролей.
- `docs/run-tests.js:1081` - `TC-15.5` проверяет настройку явного цвета и сохранение в `state.roleColors`.

**Что не покрыто:**

- rename роли при наличии явного цвета;
- delete роли при наличии явного цвета;
- move роли и влияние на fallback-палитру;
- rename роли в существующее имя;
- неизвестная роль у сотрудника при наличии/отсутствии `roleColors`.

**Почему это риск:**

Именно на rename сейчас найден самый серьезный пограничный дефект. Текущие тесты проверяют только простой happy path: "задал цвет - цвет применился".

**Рекомендация:**

Добавить отдельный блок тестов для role dictionary lifecycle:

- role color survives/changes according to выбранной семантике rename;
- delete role удаляет только ожидаемую запись `roleColors`;
- duplicate rename запрещен;
- fallback-цвет после move роли соответствует SPEC.

### F5 (P2): "скрытие грейдов" может быть понято сильнее, чем реализовано

**Где:**

- `docs/SPEC.md:288` - `ui.hideGrades` маскирует индивидуальные грейды на карточках.
- `docs/SPEC.md:289` - грейды названы чувствительной информацией.
- `index.html:657` - маскируется только квадрат грейда на карточке.

**Что происходит:**

Реализация соответствует текущей SPEC: маскируются только индивидуальные грейды на карточках, агрегаты не маскируются, подрядчики не затрагиваются. При этом значения грейдов остаются доступны:

- в localStorage;
- в JSON export;
- в модалке редактирования сотрудника;
- косвенно через агрегаты команды/кластера.

**Почему это риск:**

Если пользователь ожидает "режим приватности", текущая реализация не обеспечивает защиту данных. Если ожидается только визуальная маскировка доски для демонстрации, все нормально.

**Рекомендация:**

Уточнить формулировку SPEC: это "визуальная маскировка карточек", а не security/privacy boundary. Если нужен настоящий privacy mode, надо отдельно маскировать export, modal и, возможно, агрегаты.

### F6 (P3): color picker роли создает много undo-точек

**Где:**

- `index.html:904` - `[data-rolecolor]` вызывает `commit()` на каждое событие `input`.

**Что происходит:**

При реальном использовании color picker может генерировать много `input`-событий во время выбора цвета. Каждое событие попадает в history.

**Почему это риск:**

Undo после выбора цвета может проходить через промежуточные цвета, а не откатывать пользовательскую операцию целиком.

**Рекомендация:**

Коммитить цвет роли на `change`, либо дебаунсить `input` и создавать одну undo-точку на законченное действие.

### F7 (P3): теги не ограничены по длине и количеству

**Где:**

- `index.html:365` - нормализация тегов trim + dedup.
- `index.html:662` - рендер тегов в `.tag-pill`.

**Что хорошо:**

XSS закрыт через `esc()`, пустые строки отбрасываются, дубли удаляются.

**Что остается серой зоной:**

Очень длинные теги или большое число тегов могут раздувать карточки и ухудшать плотность доски. Отдельного layout-теста на этот случай нет.

**Рекомендация:**

Ввести продуктовые ограничения, например:

- максимальная длина одного тега;
- максимальное число тегов на сотрудника;
- CSS-ограничение для длинных токенов (`overflow-wrap`, max-width/line clamp по необходимости).

Добавить тест на длинный тег и несколько тегов.
