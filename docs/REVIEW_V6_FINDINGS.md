## Findings

### V6-F1 (P1): команда только со ставками удаляется без confirm, ставки исчезают

**Где:**

- `index.html:1058` - удаление команды из справочников (`[data-trm]`).
- `index.html:1218` - удаление команды с доски (`[data-delteam]`).
- `index.html:472` - `teamMetrics()` считает `head` без ставок и отдельно `vacancies/vacancyFte`.
- `index.html:477` - ставки исключаются из `head` через `continue`.

**Что происходит:**

Если команда содержит только открытые ставки и не содержит сотрудников, `teamMetrics(id)` возвращает:

```json
{
  "fte": 0,
  "head": 0,
  "vacancies": 1,
  "vacancyFte": 1
}
```

Удаление команды проверяет только `m.head > 0`, поэтому confirm не показывается. После удаления команды аллокация ставки снимается, ставка становится без аллокаций и удаляется как осиротевшая запись:

```js
state.people = state.people.filter(p => p.allocations.length > 0);
```

**Почему это риск:**

Ставка - рабочая сущность планирования, а не пустой placeholder. Для пользователя удаление команды со ставками без подтверждения выглядит как потеря данных. Особенно рискованно, потому что новая фича явно выводит "Открытые ставки" в шапке команды.

**Рекомендация:**

Показывать confirm, если в команде есть сотрудники или ставки:

```js
const hasContent = m.head > 0 || m.vacancies > 0;
if (hasContent && !confirm(...)) return;
```

Текст confirm должен явно упоминать оба типа:

- участников: `m.head`;
- открытые ставки: `m.vacancies`, `m.vacancyFte`.

Исправить оба пути:

- удаление с доски (`[data-delteam]`);
- удаление из справочников (`[data-trm]`).

**Тесты:**

Добавить кейс:

- fixture: одна команда, только одна ставка;
- удалить команду с доски;
- проверить, что confirm был вызван;
- при dismiss команда и ставка остаются;
- при accept команда удалена, ставка удалена/осиротела согласно текущей политике.

Отдельно повторить для удаления через settings (`[data-trm]`).

### V6-F2 (P2): импорт может создать ставку с проектом вне справочника

**Где:**

- `index.html:362` - `projects = uniqStrings(src.projects)`.
- `index.html:402` - `project = isVacancy ? asStr(p.project).trim() : ""`.
- `index.html:844` - modal select строится только из `state.projects`.
- `docs/SPEC.md:90` - `project` у ставки описан как проект из `projects`.

**Что происходит:**

Через LS/импорт можно загрузить:

```js
projects: ["Known"]
people: [
  { isVacancy: true, project: " Ghost ", ... }
]
```

После `normalizeDoc()` в памяти:

```json
{
  "isVacancy": true,
  "project": "Ghost"
}
```

На доске ставка отображается как проектная с чипом `Ghost`, но в модалке `<select id="m_project">` содержит только `— без проекта —` и `Known`. При открытии модалки выбранное значение становится пустым, то есть UI не может отредактировать/сохранить этот проект как валидный.

**Почему это риск:**

Состояние после импорта нарушает инвариант UI и SPEC: проект ставки должен быть из справочника `projects`. Это создает:

- проектные ставки, которых нет в справочнике;
- расхождение между доской и модалкой;
- неявную потерю проекта при следующем сохранении ставки;
- некорректную аналитику "проектных" ставок, если неизвестные проекты считаются проектными.

**Рекомендация:**

Выбрать политику нормализации:

1. Строгая: если `project` не пустой и отсутствует в `projects`, сбрасывать в `""`.
2. Сохраняющая: автоматически добавлять неизвестный `project` в `projects`.

Для текущей логики справочников более предсказуем вариант 1:

```js
const pr = isVacancy ? asStr(p.project).trim() : "";
const project = pr && projects.includes(pr) ? pr : "";
```

Если выбрать вариант 2, нужно тестировать дедуп/trim и rename/delete проекта для auto-added значений.

**Тесты:**

Добавить импортный/LS-кейс:

- `projects: ["Known"]`;
- ставка с `project: " Ghost "`;
- после загрузки ожидать выбранную политику:
  - strict: `person.project === ""`, чип проекта не показан;
  - preserving: `state.projects.includes("Ghost")`, select содержит `Ghost`.

### V6-F3 (P2): импорт принимает некалендарные даты ставок

**Где:**

- `index.html:401` - `expiry` проверяется регуляркой `^\d{4}-\d{2}-\d{2}$`.
- `index.html:742` - дата отображается на доске как `до ${p.expiry}`.
- `index.html:877` - в модалке используется `<input id="m_expiry" type="date">`.

**Что происходит:**

Значение `2026-99-99` проходит regex и отображается на доске:

```text
⏳ до 2026-99-99
```

Но `<input type="date">` не принимает такую дату; при открытии модалки поле срока становится пустым.

**Почему это риск:**

Импорт/LS может создать состояние, которое:

- невозможно создать через UI;
- отображается как валидный срок на доске;
- silently теряется при редактировании ставки;
- искажает аналитику временных ставок (`vacTemp` считает любую непустую `expiry`).

**Рекомендация:**

Проверять не только формат, но и календарную валидность:

```js
function validDateISO(s) {
  if (!/^\d{4}-\d{2}-\d{2}$/.test(s)) return false;
  const d = new Date(`${s}T00:00:00Z`);
  return !Number.isNaN(d.getTime()) && d.toISOString().slice(0, 10) === s;
}
```

В `normalizeDoc()`:

```js
const rawExpiry = asStr(p.expiry).trim();
const expiry = isVacancy && validDateISO(rawExpiry) ? rawExpiry : "";
```

**Тесты:**

Добавить кейс:

- импорт ставки с `expiry: "2026-99-99"`;
- ожидать `expiry === ""`;
- чип срока не отображается;
- временная ставка не учитывается в `vacTemp`.

## Grey Zones

### G1: фильтр "штат" показывает ставки

Сейчас `isVacancy` форсит `isContractor=false`, поэтому фильтр `#filterStatus = staff` оставляет ставки видимыми. Это может быть корректно, если "ставка" трактуется как штатная позиция. Но для пользователя фильтр "штат" может означать "только реальные штатные сотрудники".

Рекомендация: явно описать в SPEC/TEST_PLAN. Если нужен отдельный UX, добавить статус "Ставки" или чекбокс.

### G2: поиск по "Вакансия" не находит ставки

Сейчас поиск проверяет только `p.name`, а у ставок `name === ""`. Поэтому запрос "Вакансия" не матчится, хотя именно это слово видно на карточке. Также поиск не ищет по проекту, роли, тегам или комментарию.

Рекомендация: если ставки становятся рабочей сущностью, расширить `matchFilter()` для `isVacancy`:

- искать по тексту `"вакансия"`;
- искать по `role`, `project`, `tags`;
- комментарий лучше не включать, если он намеренно скрыт с доски.
