## Детали

### F1 (P0) — quick-add у роли реально не добавлял сотрудника

**Было:** `quickAdd(teamId, role)` передавал готовый объект в `personModal`, который определяет
режим как `const isNew = !person`. Объект → `isNew = false` → ветка сохранения:
`state.people[findIndex(...)] = p`, где `findIndex` возвращает `-1` (такого id ещё нет) →
запись по `people[-1]` (свойство, а не элемент массива). Сотрудник **не добавлялся**, длина
массива не менялась.

**Стало:**
```js
function personModal(person, defaultTeamId, defaultRole){
  const isNew = !person;
  const p = person ? clone(person) : {
    id:uid("p"), name:"", role:defaultRole||state.roles[0]||"", grade:9, isContractor:false,
    location:state.locations[0]||"", allocations: defaultTeamId?[{teamId:defaultTeamId,fte:1}]:[]
  };
  ...
}
function quickAdd(teamId, role){ personModal(null, teamId, role); }
```
Теперь quick-add открывает модалку в режиме создания с предзаполненными командой и ролью →
сохранение идёт через `push`.

**Почему ревью/тесты не ловили раньше:** TC-3.8 проверял только предвыбор роли, без сохранения.
Добавлен **TC-3.8b (P0)**: нажать «＋» у роли → заполнить имя → сохранить → проверить
`people.length+1`, роль, команду и появление `.chip` на доске.

### F2 (P1) — сводный средний грейд кластера

**Было:** `teamMetrics` корректно исключает подрядчиков из среднего грейда команды, но
аналитика затем весила `avgGrade` команды на полный `x.m.fte` (включая подрядный FTE), хотя
подпись — «взвеш. по FTE, штат».

**Стало:** `teamMetrics` возвращает дополнительное поле `gradeFte` (сумма FTE штатных с
грейдом). Аналитика весит по нему:
```js
const avgAll = (()=>{ let w=0,f=0;
  for(const x of stats.per){ if(x.m.avgGrade!=null){ w+=x.m.avgGrade*x.m.gradeFte; f+=x.m.gradeFte; } }
  return f?w/f:null; })();
```

**Покрытие — TC-8.1b (P1):** фикстура `FX_MIXGRADE` — команда X {штат г10 fte1, подрядчик
fte9}, команда Y {штат г8 fte1}. Ожидание кластерного грейда `9.0`; на старом весе было бы ~9.8.
(Грейды в диапазоне 7..12: исходный г6 был вне диапазона и клампился нормализацией — см.
заметку в TEST_REPORT.)

### F3 (P1) — валидация импорта и загрузки из localStorage

Добавлены `normalizeDoc(input)` и `normalizeSnapshots(arr)`; применяются в `load()` и
`importFile()`. Политика (подробно — SPEC §5.1):

- обязательные `meta/roles/locations/teams/people` всегда существуют; не-массивы → массивы;
- грейд парсится и клампится в `7..12`; у подрядчиков `grade=null`;
- аллокации — только с `fte>0` (парсинг из строки/числа) и существующим `teamId`;
- люди без валидных аллокаций отбрасываются;
- `baselineId` оставляется, только если указывает на существующий снимок, иначе `null`;
- после импорта в тосте — сводка (загружено команд/людей, сколько записей отброшено).

**Покрытие — TC-2.8 (P1):** импорт `FX_BADSCHEMA` (валидный JSON, кривая схема) → проверка
всех правил нормализации и отсутствия `pageerror`.

### F4 (P1/P2) — санация импортированных `id` и `color`

Закрыто у источника в `normalizeDoc`:
- `color` валидируется по `^#[0-9a-fA-F]{6}$`, иначе `#4f8cff`;
- `id` валидируется по `^[A-Za-z0-9_-]+$`; небезопасный — регенерируется (`uid`) с ремапом
  ссылок в аллокациях.

Так как значения санируются до рендера, вставка `id`/`color` в HTML-атрибуты и `style`
безопасна. Покрытие — в TC-2.8 (перегенерация небезопасного id, дефолт битого цвета).

### F5 (P2) — охват undo/redo (продуктовое решение)

Оставлено как есть: история хранит только документ `state` (состав команд, сотрудники,
справочники, имя кластера). Операции со снимками, выбор baseline и `ui.compare` под undo не
попадают — это операции над сценариями, а не над текущим составом. Поведение явно описано в
**SPEC §7.8**. Возможное расширение истории до `{state, snapshots, baselineId}` — в бэклоге §10.

### F6 (P2) — мёртвый CSS `.chip.moved-out`

Класс нигде не выставлялся (в режиме сравнения ставится только `.moved-in`). Удалён из CSS;
карта DOM в SPEC §8 и TEST_PLAN §3 приведена в соответствие.

### Предложение — предупреждение о суммарном FTE > 1

Сумма FTE по человеку намеренно не ограничена. Добавлено мягкое предупреждение в модалке
редактирования (строка `#m_allocsum`): при сумме > 1.0 текст окрашивается в warning-цвет и
дополняется пометкой «⚠ больше 1.0 (перегрузка)». Запрета нет.

## V2-F1 (P1) — дубли безопасных id

**Было:** `normalizeDoc` регенерировал id только при провале `SAFE_ID`, но не проверял
уникальность. Два объекта с `id:"tA"` оставались с одинаковым id → одинаковый `data-team-id`,
`teamMetrics` считает аллокации обеих, удаление/rename по `find` бьёт в первую, DnD/compare по
людям с общим id неоднозначны.

**Стало:** в `normalizeDoc` добавлены `seenTeam`/`seenPerson`, в `normalizeSnapshots` — `seen`.
Если id безопасный, но уже занят в документе → регенерация через `uid` (с гарантией уникальности
через `do…while`). Политика для команд: первый дубликат сохраняет id (аллокации на него уходят к
нему), последующие получают новый. Покрытие — **TC-2.9** (дубли team/person/snapshot id →
уникальность после нормализации, без `pageerror`).

## V2-F2 (P1/P2) — почему не делаем общий ремап (обоснование)

`state` и каждый `snapshot.doc` нормализуются независимо. Это **проблема только для рукотворного
импорта**, где одни и те же *небезопасные/дублирующиеся* id есть и в `state`, и в `snapshot.doc`:
их независимая регенерация может разойтись → compare с таким снимком ложно подсветит всех как
`moved-in`.

Почему оставлено и задокументировано, а не «починено» общим `idMap`:
- **Данные приложения никогда не триггерят баг.** Экспорт всегда содержит безопасные уникальные
  id; для них `normalizeDoc` идемпотентен → после round-trip импорт/экспорт `state` и снимки
  сохраняют те же id, compare корректен. Это зафиксировано тестом **TC-12.8**.
- **Наивный общий `seen`-set сломал бы нормальный путь:** одинаковый безопасный id в `state` и в
  `snapshot.doc` — это *одна и та же* сущность (должна сохранить id), а не коллизия. Кросс-док
  «дедуп» регенерировал бы id снимка и сам сломал бы compare для реальных данных.
- Корректное решение (общий `idMap` только для небезопасных id, при сохранении пер-документной
  дедупликации дублей) сложнее и рискованнее самой проблемы с near-zero вероятностью.

Итог: ограничение явно описано в **SPEC §5.1** (раздел «Граница») и внесено в бэклог **§10**.

## V3-F1 (P1) — семантика rename роли

**Было:** rename переносил `roleColors[old]→[new]`, но `people[].role` оставался старым →
существующие сотрудники теряли явный цвет (падали на fallback). Два правила конфликтовали.

**Стало (продуктовое решение — настоящее переименование):** обработчик `[data-roleedit]`
мигрирует `people[].role` со старого значения на новое **и** переносит цвет. Справочник и
сотрудники остаются согласованы, «осиротевших» значений не возникает. Та же логика применена к
переименованию **локаций** (`[data-edit]` → миграция `people[].location`). SPEC §7.6/§7.14
и подсказка в справочнике обновлены. Покрытие — **TC-16.1**.

## V3-F2 (P1) — запрет дублей при rename

**Было:** rename принимал любое непустое имя; роль можно было переименовать в уже существующую →
два одинаковых имени в `state.roles` (неоднозначность `roleColors`, fallback-индекса, опций в UI).

**Стало:** rename в имя, которое уже есть (кроме самой себя), отклоняется — поле возвращает
прежнее значение + toast «уже существует». Аналогично для локаций. Покрытие — **TC-16.2**.

## V3-F3 (P2) — усиление TC-15.3

порядок) → **redo** (новый) → **reload** (порядок сохранён). Раньше проверялся только DOM/LS.

## V3-F4 (P2) — lifecycle `roleColors` (блок TS-16)

Покрытие — **TC-16.6**.
