# Team Balancer — результаты ревью

Дата ревью: 2026-06-30.

Цель документа: передать агенту-разработчику конкретные дефекты, пробелы тестов и предложения по требованиям после чтения `docs/SPEC.md`, `docs/TEST_PLAN.md`, `docs/TEST_REPORT.md`, `docs/run-tests.js` и `index.html`.

Код приложения в ходе ревью не изменялся. Быстрая проверка синтаксиса встроенного `<script>` прошла успешно: `script syntax OK`. Полный Playwright-прогон не запускался, потому что пакет `playwright` не установлен в текущей папке.

## Краткий вывод

Проект в целом хорошо соответствует спецификации: однофайловое приложение без сетевых зависимостей, с локальным хранением, JSON-импортом/экспортом, DnD, снимками, фильтрами, undo/redo и автотестовым раннером.

Основные риски:

- есть минимум один вероятный функциональный баг в добавлении сотрудника через кнопку `＋` у роли;
- часть расчётов аналитики может быть математически неверной в смешанных командах со штатом и подрядчиками;
- импорт и загрузка из `localStorage` доверяют форме данных почти без нормализации;
- тесты покрывают широкий happy path, но некоторые кейсы проверяют только наличие эффекта, а не полную инвариантность данных.

## Findings

### P0/P1: добавление через `＋` у роли не сохраняет нового сотрудника корректно

Файлы:

- `index.html:925-928` — `quickAdd(teamId, role)`
- `index.html:616-620` — `personModal(person, defaultTeamId)`
- `index.html:700-701` — ветка сохранения в модалке

Проблема:

`quickAdd` вызывает `personModal` с готовым объектом сотрудника:

```js
function quickAdd(teamId, role){
  personModal({id:uid("p"),name:"",role,grade:9,isContractor:false,
    location:state.locations[0]||"",allocations:[{teamId,fte:1}]});
}
```

`personModal` определяет режим по факту наличия аргумента:

```js
const isNew=!person;
```

То есть объект из `quickAdd` считается существующим сотрудником. При сохранении выполняется ветка редактирования:

```js
else { const idx=state.people.findIndex(x=>x.id===p.id); state.people[idx]=p; }
```

Такого `id` в `state.people` нет. `findIndex` вернёт `-1`, и сохранение пойдёт в некорректную позицию массива вместо обычного `push`.

Почему тесты не поймали:

`TC-3.8` проверяет только то, что роль предвыбрана в модалке. Он не нажимает `Сохранить` и не проверяет появление нового сотрудника на доске/в `localStorage`.

Рекомендация:

- Изменить `quickAdd` так, чтобы модалка открывалась в режиме создания, но с предзаполненной ролью и командой.
- Один простой вариант: расширить `personModal(null, defaultTeamId, defaultRole)` или добавить отдельный фабричный параметр `defaults`.
- Добавить тест: нажать `＋` у роли, заполнить имя, сохранить, проверить `state.people.length + 1`, роль, команду и наличие `.chip`.

### P1: сводный средний грейд кластера может считаться с неверным весом

Файлы:

- `index.html:329-344` — `teamMetrics`
- `index.html:432-446` — `renderAnalytics`

Проблема:

`teamMetrics` правильно исключает подрядчиков из среднего грейда команды:

```js
else if(p.grade!=null){ gradeW+=p.grade*a.fte; gradeFte+=a.fte; }
```

Но сводный средний грейд в аналитике затем весит `avgGrade` команды на общий FTE команды:

```js
if(x.m.avgGrade!=null){w+=x.m.avgGrade*x.m.fte;f+=x.m.fte;}
```

`x.m.fte` включает подрядчиков. Если в команде 1 штатный FTE с грейдом 10 и 9 подрядных FTE, командный `avgGrade` 10 получит вес 10, хотя подпись говорит `взвеш. по FTE, штат`.

Рекомендация:

- Вернуть из `teamMetrics` отдельное поле `gradeFte`.
- В `renderAnalytics` считать `w += avgGrade * gradeFte`, `f += gradeFte`.
- Добавить тест с двумя командами, где одна команда содержит много подрядчиков, и проверить ожидаемое значение среднего грейда кластера.

### P1: импорт и загрузка из localStorage почти не валидируют схему

Файлы:

- `index.html:303-310` — `load`
- `index.html:826-837` — `importFile`

Проблема:

`load()` принимает любой JSON с `state`:

```js
state=d.state;
```

`importFile()` проверяет только наличие `d.state.teams` или `d.teams && d.people`:

```js
if(d.state && d.state.teams){ state=d.state; snapshots=d.snapshots||[]; baselineId=d.baselineId||null; }
else if(d.teams && d.people){ state=d; }
```

Валидный JSON неправильной формы может сломать рендер или доменную логику: отсутствующие `roles`, `locations`, `people`; `allocations` не массив; `fte` строкой/NaN; аллокации на несуществующие команды; `grade` вне диапазона; contractor с ненулевым grade.

Рекомендация:

- Добавить функцию нормализации/валидации документа, например `normalizeDoc(input)`.
- Минимально гарантировать:
  - `meta`, `roles`, `locations`, `teams`, `people` существуют;
  - `teams` имеют безопасные `id`, `name`, `color`;
  - `people` имеют безопасные `id`, `name`, `role`, `location`, `isContractor`, `grade`;
  - `allocations` только с существующими `teamId` и `fte > 0`;
  - `grade` клампится, у подрядчиков `grade=null`;
  - люди без валидных аллокаций либо отбрасываются, либо получают дефолтную аллокацию по явному правилу.
- Добавить тесты на валидный JSON неправильной схемы, а не только на битый JSON.

### P1/P2: импортированные `id` и `color` вставляются в HTML/атрибуты без валидации

Файлы:

- `index.html:485` — `style="background:${team.color...}"`
- `index.html:735-737` — цвет и `data-tname`
- `index.html:781-788` — `data-base`, `data-restore`, `data-export`, `data-delsnap`

Проблема:

Для обычных данных, созданных приложением, `uid()` и `input[type=color]` делают значения безопасными. Но импортированный JSON может содержать произвольные `id` и `color`.

Примеры рискованных мест:

```html
<span class="team-color" style="background:${t.color||'#4f8cff'}"></span>
<input value="${esc(t.name)}" data-tname="${t.id}">
<button class="btn sm" data-base="${s.id}">
```

`esc()` применяется к именам, но не к `id` и цветам. При текущей модели данных импорт считается пользовательским входом, значит эти поля тоже нужно нормализовать.

Рекомендация:

- Валидировать цвет регуляркой `^#[0-9a-fA-F]{6}$`, иначе ставить дефолт.
- Для импортированных `id` либо:
  - принимать только безопасный формат вроде `/^[a-zA-Z0-9_-]+$/`;
  - либо регенерировать внутренние id и перепривязать аллокации/снимки.
- При вставке в HTML-атрибуты использовать отдельный `escAttr`, если значения остаются произвольными.

### P2: undo/redo покрывает только `state`, но не снимки, baseline и часть UI

Файлы:

- `index.html:316-324` — история
- `index.html:803-811` — операции со снимками

Наблюдение:

Спецификация прямо говорит, что `History (undo/redo)` хранит снимок документа `state`. Поэтому это не баг относительно текущего текста. Но для пользователя операции со снимками и базой тоже выглядят как изменения документа/сценария.

Сейчас undo:

- откатывает переносы, сотрудников, команды, справочники;
- не откатывает создание/удаление снимка;
- не откатывает выбор baseline;
- не восстанавливает `ui.compare`.

Рекомендация:

Решить продуктово:

- оставить как есть и явно написать в спецификации, что undo работает только с текущим составом команд;
- либо расширить историю до `{state, snapshots, baselineId}` и добавить тесты на undo удаления/создания снимка и выбора базы.

### P2: `.moved-out` описан в DOM-карте, но фактически не используется

Файлы:

- `index.html:124` — стиль `.chip.moved-out`
- `docs/TEST_PLAN.md:102` — класс указан в DOM-карте
- `index.html:534-550` — `renderChip` ставит только `.moved-in`

Наблюдение:

Спецификация в разделе 7.7 требует подсветку переехавших участников в новых командах (`moved-in`) и дельты по командам. `moved-out` не заявлен как обязательное поведение, но класс есть в CSS и тестовой DOM-карте.

Рекомендация:

- Либо удалить `moved-out` из DOM-карты/CSS как неиспользуемый остаток.
- Либо реализовать визуализацию ушедших из команды людей в режиме сравнения, если это полезно для as-is/to-be.

## Пробелы тестового покрытия

### Не покрыто сохранение через quick-add у роли

См. finding про `quickAdd`. Добавить тест поверх `TC-3.8`.

### Не покрыта точная формула сводного среднего грейда

`TC-8.1` проверяет значение на простой фикстуре, где ошибка с весом подрядчиков может не проявляться. Нужна фикстура с большим подрядным FTE в команде, где есть штатный грейд.

### Не покрыты структурно некорректные импорты

Сейчас есть тест на битый JSON (`TC-2.6`) и happy path для полного/голого документа. Нужны кейсы:

- `roles`/`locations` отсутствуют;
- `people` отсутствует или не массив;
- `allocations` отсутствуют или не массив;
- аллокации с неизвестным `teamId`;
- `fte` строкой, `NaN`, отрицательный, слишком большой;
- contractor с `grade`;
- duplicate ids.

### Не полностью покрыты кейсы, отмеченные в отчёте как представительские

`docs/TEST_REPORT.md` указывает, что не все низкоприоритетные кейсы покрыты дословно:

- `TC-7.4` — флаг по доле подрядчиков;
- `TC-7.6` — команд меньше двух;
- `TC-9.5` — серия нескольких undo;
- `TC-10.5` — комбинация фильтров;
- `TC-11.5` — rename/delete локаций.

Стоит либо расширить раннер, либо явно понизить требования тест-плана до representative coverage.

### `TC-14.4` есть в плане, но не найден в раннере

Тест-план содержит кейс про очень длинное имя и 12+ команд. В `docs/run-tests.js` такого теста не видно. Нужно добавить или удалить/перенести из обязательного списка.

### PNG считается pass при фолбэке

Это соответствует спецификации, но означает, что реальная генерация PNG почти не гарантируется. Если PNG важен как функция, нужен отдельный ручной/браузерный чек в headed Chromium/Edge.

## Предложения к требованиям

### Добавить требования к валидации данных

Сейчас спецификация описывает модель данных, но не описывает, что делать с плохим импортом. Рекомендуется добавить раздел:

- поддерживаемая версия формата;
- обязательные поля и дефолты;
- нормализация старых/частичных документов;
- политика по некорректным людям/аллокациям;
- сообщение пользователю после импорта: сколько записей загружено, сколько исправлено/отброшено.

### Добавить предупреждение о суммарном FTE человека > 1

Спецификация намеренно разрешает сумму FTE > 1. Это нормально для гибкости, но пользователю полезно видеть предупреждение, а не только сумму.

Вариант:

- `<=1.0` — обычный цвет;
- `>1.0` — warning в модалке и на карточках/аналитике;
- запрета нет.

### Добавить целевые эталоны по командам

Уже есть в бэклоге. Это сильное развитие продукта:

- целевой FTE команды;
- минимальный/целевой средний грейд;
- максимальная доля подрядчиков;
- обязательные роли;
- дельта к целевому состоянию.

### Добавить рольное покрытие

Сейчас баланс учитывает грейд и подрядчиков. Для IT-лидера часто важнее увидеть дыры по ролям:

- нет QA;
- нет BA/SA;
- один Backend без резервирования;
- отсутствует Data Engineer в командах, где роль нужна.

Это можно сделать как отдельную метрику без нарушения текущей модели.

### Добавить явный статус приватности

Приложение уже локальное и без сетевых запросов. Можно добавить в спецификацию и тесты:

- статическая проверка отсутствия `fetch`, `XMLHttpRequest`, `sendBeacon`, внешних `<script src>`, `<link href>`, `@import`;
- запрет внешних URL кроме XML namespace SVG и локального тестового сервера в документации.
